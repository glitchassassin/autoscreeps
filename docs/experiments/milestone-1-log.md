### Hypothesis

- Before changing the opener, we need a fixed-suite baseline that tells us whether source utilization, spawn idling, and early RCL timing are stable enough to optimize against.

- Cases and run IDs:
  - `train-a-2k`: `2026-04-04T19-07-06-856Z-7abb1848`
  - `train-b-2k`: `2026-04-04T19-08-49-543Z-9dd62b55`
  - `train-c-5k`: `2026-04-04T19-10-33-534Z-b6fb57f5`

  - `holdout-b-5k`: `2026-04-04T19-26-14-585Z-9612a004`

### Results

- Suite completion: `6/6` cases completed.
- Baseline-side median primary metrics:
  - `T_RCL2`: `546.5`
  - `T_RCL3`: not reached in any case
  - `spawnIdlePct`: `13.19%`
  - `sourceCoveragePct`: `50.70%`
  - `sourceUptimePct`: `1.41%`
- Notable per-case behavior:
  - `train-b-2k` was much healthier than the other cases, with `sourceCoveragePct 95.14%` and `sourceUptimePct 90.28%`.
  - Several other maps hovered near `50%` source coverage, which is consistent with only one source being effectively staffed for much of the run.
  - No case reached `RCL3` by `5000` ticks.
- Notes on interpretation of the suite summary:
  - Because the run compared `workspace` against itself, any baseline-vs-candidate differences are mirror/run variance, not a real code delta.
  - The useful part of this run is the absolute baseline numbers and the map-to-map variance.

### Interpretation

- The current opener is stable enough to benchmark against, but it is not reliably utilizing both sources.
- The likely causes are:
  - fixed `2 harvester / 2 upgrader` staffing regardless of source coverage
  - `findClosestByPath` source selection with no sticky assignment or balancing
  - upgraders competing directly for raw source access during bootstrap
- The strong result on `train-b-2k` suggests the problem is inconsistency rather than a universally weak economy.
- Hypothesis supported: the benchmark made source-utilization inconsistency visible enough to target directly.

### Follow-Up Hypotheses

- Hypothesis A: sticky balanced source assignment will improve `sourceCoveragePct`, `sourceUptimePct`, and `T_RCL2` on the weak maps.
- Hypothesis B: coverage-first spawn priorities will reduce time spent with an uncovered source and improve early controller timing.
- Hypothesis C: throttling early upgraders until both sources are staffed will reduce source contention and produce a more stable opener.
- Hypothesis D: if the above are not enough, the opener will need true role separation with dedicated miners and hauling.

### Decision

- `iterate`
- Keep this benchmark as the milestone-one baseline and start with the smallest source-assignment experiment before attempting a larger economy rewrite.

## Entry `exp-2026-04-04-sticky-balanced-source-assignment`

- Status: `completed`
- Owner: `OpenCode`
- Date: `2026-04-04`

### Hypothesis

- If creeps keep a balanced source assignment instead of choosing the nearest source every harvest cycle, weak maps should spend less time with one uncovered source.
- That should improve `sourceCoveragePct`, `sourceUptimePct`, and likely `T_RCL2` without requiring a larger role or logistics rewrite.

### Experiment

- Variant or branch: `git:HEAD` vs `workspace`
- Bot package: `bots/basic`
- Suite: `experiments/suites/milestone-1-opener.yaml`
- Change tested:
  - make `sourceId` a sticky assignment rather than a passive observation
  - assign new harvesters and upgraders to the least-loaded source in their room
  - keep the existing role model and body plans unchanged
- Cases and run IDs:
  - `train-a-2k`: `2026-04-04T21-50-31-586Z-a3120f19`
  - `train-b-2k`: `2026-04-04T21-59-33-836Z-78b1d16e`
  - `train-c-5k`: `2026-04-04T22-08-35-944Z-544707f0`
  - `train-d-5k`: `2026-04-04T22-11-56-199Z-009d42ad`
  - `holdout-a-2k`: `2026-04-04T22-33-29-873Z-06635380`
  - `holdout-b-5k`: `2026-04-04T22-36-24-493Z-10ec9db9`

### Results

- All `6/6` cases completed.
- Train cohort summary:
  - `T_RCL2`: regressed from `577` to `581.5` (`+0.78%`)
  - `spawnIdlePct`: improved from `12.44%` to `11.54%`
  - `sourceCoveragePct`: improved from `50.81%` to `97.23%`
  - `sourceUptimePct`: improved from `1.62%` to `94.47%`
- Holdout cohort summary:
  - `T_RCL2`: regressed from `464` to `490` (`+5.6%`)
  - `spawnIdlePct`: regressed from `11.13%` to `12.12%` (`+8.89%`)
  - `sourceCoveragePct`: improved from `50.99%` to `97.46%`
  - `sourceUptimePct`: improved from `1.98%` to `94.93%`
- Gate result:
  - training passed with `3` improved primary metrics
  - holdout failed because `T_RCL2` regressed by more than `5%` and `spawnIdlePct` regressed by more than `5%`
  - overall suite result: `failed`

### Interpretation

- Sticky source assignment did not produce a promotable opener.
- The large jumps in `sourceCoveragePct` and `sourceUptimePct` are not credible on their own, because those metrics now reflect persistent source assignment much more than active harvesting.
- The strongest evidence comes from the real progress metric: `T_RCL2` did not improve and regressed on holdout.
- Likely explanation:
  - assignment bookkeeping improved, but the opener still has direct-source contention from upgraders
  - sticky assignment without active rebalancing can keep creeps tied to a source even when that is not the best short-term choice
  - the current source telemetry needs an active-harvest signal before it can be trusted as the main decision metric for this class of experiment
- Hypothesis not supported as a standalone promotable change.

### Follow-Up Hypotheses

- Hypothesis A: add telemetry that distinguishes assigned sources from actively harvested sources before attempting another source-utilization experiment.
- Hypothesis B: the next meaningful opener test should be coverage-first spawning or early upgrader throttling, not assignment alone.
- Hypothesis C: if sticky assignment is revisited, it should include explicit rebalance rules when a creep's assigned source is not actively improving throughput.

### Decision

- `reject`
- Do not promote this change. Keep the baseline bot behavior, keep the result in the log, and improve the source instrumentation before trusting source-coverage deltas in future experiments.

## Entry `exp-2026-04-04-assignment-vs-harvest-mode-telemetry`

- Status: `completed`
- Owner: `OpenCode`
- Date: `2026-04-04`

### Hypothesis

- If the sticky-assignment result was mostly a telemetry artifact, then assignment-based source coverage should jump sharply while harvest-mode coverage remains much lower.
- If that happens alongside weak or regressed `T_RCL2`, we can reject the sticky-assignment change with much more confidence.

### Experiment

- Variant or branch: `git:HEAD` vs `workspace`
- Bot package: `bots/basic`
- Suite: `experiments/suites/milestone-1-opener.yaml`
- Change set:
  - keep the rejected sticky-assignment candidate only long enough to rerun it under improved telemetry
  - add source telemetry that distinguishes assigned sources from sources currently used by creeps in harvest mode
- Cases and run IDs:
  - `train-a-2k`: `2026-04-04T22-53-12-355Z-e9f1ad6c`
  - `train-b-2k`: `2026-04-04T22-54-57-692Z-c332b8c9`
  - `train-c-5k`: `2026-04-04T22-56-41-377Z-40d50a04`
  - `train-d-5k`: `2026-04-04T23-00-01-414Z-0029bbf2`
  - `holdout-a-2k`: `2026-04-04T23-03-22-740Z-b8e33bd4`
  - `holdout-b-5k`: `2026-04-04T23-12-24-559Z-0019c3af`

### Results

- All `6/6` cases completed.
- Suite gate result:
  - training passed with improvements in `spawnIdlePct`, `sourceCoveragePct`, and `sourceUptimePct`
  - holdout failed because `T_RCL2` regressed by `5.97%` and `spawnIdlePct` regressed by `8.49%`
  - overall result: `failed`
- Candidate-side assignment vs harvest-mode medians:
  - overall:
    - assigned source coverage: `97.37%`
    - harvest-mode source coverage: `60.31%`
    - assigned source uptime: `94.74%`
    - harvest-mode source uptime: `35.88%`
  - train:
    - assigned source coverage: `97.37%`
    - harvest-mode source coverage: `62.86%`
    - assigned source uptime: `94.74%`
    - harvest-mode source uptime: `42.40%`
  - holdout:
    - assigned source coverage: `97.51%`
    - harvest-mode source coverage: `55.84%`
    - assigned source uptime: `95.03%`
    - harvest-mode source uptime: `32.05%`
- Progress metrics still did not improve enough:
  - overall `T_RCL2`: regressed from `553` to `565.5`
  - holdout `T_RCL2`: regressed from `460.5` to `488`

### Interpretation

- This confirms the earlier suspicion: a large portion of the previous source-coverage jump was a telemetry artifact caused by treating sticky assignment as if it were active source utilization.
- The candidate kept almost every source assigned on paper, but the stronger harvest-mode metric shows that only about half to two-thirds of sources were actively covered at sample time.
- The gap is especially clear on holdout:
  - assignment coverage near `97.5%`
  - harvest-mode coverage only `55.84%`
  - assignment uptime near `95%`
  - harvest-mode uptime only `32.05%`
- Since `T_RCL2` and holdout `spawnIdlePct` still regressed, the sticky-assignment change did not produce a real opener improvement that survived a better metric.
- Hypothesis supported: the earlier result was mostly an assignment-telemetry artifact rather than a clear economic win.

### Follow-Up Hypotheses

- Hypothesis A: coverage-first spawn priorities are a better next experiment than source assignment alone.
- Hypothesis B: early upgrader throttling until both sources are actively covered may reduce source contention more effectively than sticky assignment.
- Hypothesis C: future source-utilization experiments should use harvest-mode metrics, not assignment-only metrics, as the primary decision signal.

### Decision

- `reject`
- Keep the harvest-mode telemetry refinement, but do not keep the sticky-assignment behavior change.
